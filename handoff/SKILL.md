# Handoff Skill

Captures the current state of a project or conversation into a structured `HANDOFF.md` file. Creates the file if it doesn't exist; intelligently merges into it if it does.

---

## When to trigger

- User says: "create a handoff", "update the handoff", "write HANDOFF.md", "save my progress", "hand this off", "I'm done for now, save state", "create a handoff doc"
- User is wrapping up a session and wants continuity
- User is passing work to another person or AI agent

---

## Step 1 — Gather context

Before writing anything, extract answers to these five questions from:
1. **The current conversation** (read carefully — most context lives here)
2. **Any existing `HANDOFF.md`** (check the working directory; if found, load it)
3. **Ask the user** only for gaps that can't be inferred

### Five questions to answer:

| Section | Question |
|---|---|
| **Goal** | What is the overall objective? What does "done" look like? |
| **Current Progress** | What has been completed so far? What is the current state? |
| **What Worked** | Which approaches, tools, or decisions produced good results? |
| **What Didn't Work** | What was tried and failed, and why? (Critical — prevents repeated mistakes) |
| **Next Steps** | What are the concrete, prioritised actions needed to continue? |

**Inference rules:**
- Infer the Goal from the earliest or most prominent user intent in the conversation
- Infer Current Progress from what has been completed or produced in the session
- Infer What Worked from approaches Claude used that the user accepted or praised
- Infer What Didn't Work from rejected attempts, errors, or explicit user corrections
- Infer Next Steps from open threads, incomplete tasks, or user's stated intentions

If fewer than 3 of the 5 sections can be confidently filled from context, ask the user one focused question before proceeding.

---

## Step 2 — Check for existing HANDOFF.md

```bash
# Check if file exists in current working directory
ls HANDOFF.md 2>/dev/null && echo "EXISTS" || echo "NEW"
```

- **If NEW**: create from scratch using the template below
- **If EXISTS**: load the file, then **merge** — do not overwrite prior content

### Merge rules (for existing files):
- **Goal**: Replace only if the user has explicitly stated a new or refined goal; otherwise keep as-is
- **Current Progress**: Append new progress below existing entries with a timestamp or session marker
- **What Worked**: Append new entries; never remove old ones
- **What Didn't Work**: Append new entries; never remove old ones (institutional memory)
- **Next Steps**: Replace entirely with the updated list — old next steps are now either done or superseded

---

## Step 3 — Write or update HANDOFF.md

Use this exact structure:

```markdown
# HANDOFF

> Last updated: {DATE} {TIME}

---

## 🎯 Goal

{Clear statement of what we're trying to accomplish and what success looks like.}

---

## ✅ Current Progress

{Bullet list of what has been completed. Be specific — include file names, decisions made, outputs produced.}

- 
- 

---

## 💡 What Worked

{Approaches, tools, patterns, or decisions that succeeded. Future sessions should lean on these.}

- 
- 

---

## ❌ What Didn't Work

{Approaches that failed, with a brief note on why. Do not repeat these.}

- 
- 

---

## 🔜 Next Steps

{Ordered list of concrete actions to continue the work. Most important first.}

1. 
2. 
3. 

---

*Generated by the handoff skill. Update this file at the end of each session.*
```

### Formatting rules:
- Use **bullet points** for Progress, Worked, Didn't Work — one idea per bullet
- Use **numbered list** for Next Steps — order by priority
- Be **specific and concrete** — avoid vague entries like "worked on the code"
- Include **file names, function names, command snippets** where relevant
- Keep each entry to 1–2 lines max; this is a reference doc, not a report

---

## Step 4 — Save the file

```bash
# Write to HANDOFF.md in the current working directory
cat > HANDOFF.md << 'EOF'
{content}
EOF
```

After saving, confirm to the user:
- Whether the file was **created** or **updated**
- A brief 2–3 line summary of what was captured
- Where the file was saved

Do **not** print the full file contents in chat — just confirm it was saved and offer to show a section if they want.

---

## Edge cases

| Situation | Behaviour |
|---|---|
| No conversation context at all | Ask the user to briefly describe the goal and what's been done |
| User provides partial info | Fill what you can, mark missing sections with `_Not captured — add manually._` |
| Multiple projects in one conversation | Ask which project this handoff is for before writing |
| User wants to see the file after saving | Run `cat HANDOFF.md` and display it |
| User asks to reset/start fresh | Confirm before overwriting; offer to archive the old file as `HANDOFF.{timestamp}.md` |
| No filesystem access (chat-only context) | Produce the HANDOFF.md content as a markdown artifact the user can copy |

---

## Output when done

End with a short confirmation like:

> ✅ `HANDOFF.md` updated — captured [N] progress items, [N] next steps. Ready to resume anytime.

Keep it one line. No lengthy summaries.
